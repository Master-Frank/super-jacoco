# 覆盖率累积更新方案（支持代码更新后继续累计）

## 1. 背景与目标

### 背景

现有覆盖率结果以“单次任务一次性产出”为主：每次执行生成一次覆盖率报告，无法把多次测试产生的覆盖增量合并为同一份可持续演进的覆盖率结果。同时，被测服务会发生代码更新并重新部署，历史覆盖数据必须在代码变更后失效，避免“旧覆盖误算到新代码”。

### 目标

- 支持“多次测试增量覆盖合并”：同一部署版本下，多次采集/执行的覆盖命中做合并，覆盖率随测试补齐逐步提升。
- 支持“代码更新后覆盖失效”：部署产物变化后，旧覆盖不再计入新版本，新版本从 0% 重新累计。
- 对前端友好：覆盖率明细通过 API 读取结构化数据渲染，保证页面打开速度与可扩展性。

## 2. 现状与差距

### 现状结论

- 平台语义上不支持“同一份结果持续更新”：Unit 入口禁止同 uuid 重复触发，见 [CodeCovServiceImpl.java:L276-L296](file:///d:/project/super-jacoco/src/main/java/com/xiaoju/basetech/service/impl/CodeCovServiceImpl.java#L276-L296)。
- Env/Local 出报仅使用当前一次 dump 获取的 `jacoco.exec` 生成结果，见 [CodeCovServiceImpl.java:L939-L984](file:///d:/project/super-jacoco/src/main/java/com/xiaoju/basetech/service/impl/CodeCovServiceImpl.java#L939-L984)。

### 差距本质

覆盖率“累积更新”与“代码更新后失效”要求平台具备两个能力：

1) **合并多次采集数据**：需要把多个 `jacoco.exec` 合并为一个 merged.exec，再生成统一的覆盖率明细。
2) **绑定到字节码产物**：覆盖率只能在同一份 class 字节码上累积；一旦字节码变化，旧覆盖必须失效。

## 3. 跨 commit 累计方案

跨 commit 累计的关键点不是合并 exec，而是**把覆盖结果转成“源码级快照”，再基于 Git diff 做映射与继承**。核心思想：

- **每个 commit 的覆盖结果先解析成结构化 JSON/快照**（按文件/方法/行）
- **当 commit 变化时**，用 Git diff 识别“未变更的代码行”，把旧覆盖继承到新 commit；对变更行清零（未覆盖）
- **新 commit 的新测试覆盖**再与继承后的快照做并集，形成“跨 commit 的累计结果”

### 3.1 总体设计

引入三层概念，保证“跨 commit 继承、变更行清零”：

- **campaign（覆盖活动，可选）**：同一服务、同一分支、同一统计口径的长期覆盖活动。
- **coverage_set（覆盖集合）**：绑定到“服务 + 分支 + 口径”，允许跨 commit 累计。
- **coverage_run（覆盖采集）**：一次 commit 的一次测试产出（exec 或 xml），附带 `commit_id`。

核心规则：

- coverage_set 不再绑定单个构建指纹，而是绑定 **服务/分支/口径**。
- 每次新 commit 进入 set 时，先进行 **diff 继承**，只继承未变更行的覆盖。
- 变更行、新增行的覆盖必须由新 commit 的测试补齐。

### 3.2 关键机制：源码级覆盖快照 + diff 继承

#### 1) 覆盖快照（Coverage Snapshot）

每次 run 完成后，从 `jacoco.xml` 解析出“源码级覆盖快照”，存为结构化数据：

- 维度：`commit_id + file_path + line_no`
- 指标：`covered/missed/hits`
- 辅助：`line_hash`（该行源码内容 hash，用于跨 diff 精确映射）

#### 2) diff 继承（核心）

当出现新 commit 时：

- 使用 Git diff（`--find-renames`）得到旧 commit -> 新 commit 的行级映射
- 继承规则：
  - **未变更行**：继承旧覆盖状态
  - **变更行 / 新增行**：标记为未覆盖
  - **删除行**：丢弃
- 对于“行号移动但内容不变”的情况，通过 `line_hash` 辅助二次匹配，减少误判

#### 2.1) diff 映射算法示意图（行级继承）

```
旧 commit (A)                         新 commit (B)
src/Foo.java                          src/Foo.java
1: package ...                        1: package ...              (未变更，继承覆盖)
2:                                   2:
3: public class Foo {                 3: public class Foo {        (未变更，继承覆盖)
4:   int a = 1;                       4:   int a = 2;              (内容变更，清零)
5:   void f() {                       5:   void f() {              (未变更，继承覆盖)
6:     doOld();                       6:     doNew();              (内容变更，清零)
7:   }                                7:   }
8: }                                  8: }

继承规则：
- diff 标记为 unchanged 的行 -> 继承覆盖
- diff 标记为 modified/added 的行 -> 标记未覆盖
- diff 标记为 deleted 的行 -> 丢弃
- diff 标记为 moved 但 line_hash 相同 -> 继承覆盖
```

#### 2.2) 方法级继承规则示意图

```
旧 commit (A)                                  新 commit (B)
class Foo {                                    class Foo {
  void f(int x) {                                void f(int x) {
    doOld();                                     doNew();            (方法体变更 -> 清零)
  }                                            }

  void g() {                                    void g() {           (未变更 -> 继承)
    ok();                                       ok();
  }                                            }

  void h() {                                    // h 删除             (删除 -> 丢弃)
    legacy();                                   void k() {           (新增 -> 清零)
  }                                              add();
}                                              }

继承规则：
- 方法签名相同且方法体 diff 为 unchanged -> 继承覆盖
- 方法签名相同但方法体 diff 为 modified -> 清零
- 方法删除 -> 丢弃
- 方法新增 -> 清零
- 方法重命名 -> 需要 rename 检测或基于 AST/内容相似度匹配，否则按“删除+新增”处理
```

#### 3) 合并新覆盖

- 新 commit 测试产生的覆盖结果与“继承快照”做并集（covered OR covered）
- 生成该 commit 的“累计覆盖快照”

### 3.3 为什么不能直接合并 exec

exec 是字节码探针命中数据，跨 commit 字节码变化时探针位置可能变化，直接合并会导致覆盖错位和虚高。跨 commit 累计必须在**源码语义层**完成。

### 3.4 数据流（端到端）

1) 采集阶段（每次测试/每次触发）

- 产出 `jacoco.exec` 并生成本次 `jacoco.xml.gz`
- 解析 xml，生成 “commit 级覆盖快照”（按文件/行/命中数）

2) 继承阶段（commit 变化时）

- 获取上一个 commit 的累计快照
- 使用 Git diff 计算旧 commit -> 新 commit 的行映射
- 生成“继承快照”（未变更行继承，变更行清零）

3) 合并阶段（生成累计结果）

- 将新 commit 的测试覆盖与继承快照做并集
- 得到该 commit 的“累计覆盖快照”
- 仅保留最新 commit 的累计快照，覆盖写入，不保留历史快照
- 可选：汇总 Package/Class/Method 结构化统计供前端展示

生成 xml 可复用现有 in-process 报告生成器（已支持输出 `jacoco.xml`）：[JacocoInProcessReportGenerator.java:L161-L219](file:///d:/project/super-jacoco/src/main/java/com/xiaoju/basetech/util/JacocoInProcessReportGenerator.java#L161-L219)。

### 3.5 正确性保证

- **不会把旧覆盖算到新代码**：只继承 Git diff 判定为“未变更”的行。
- **覆盖提升可解释**：新 commit 覆盖 = 继承快照（未变更行） + 新测试命中。
- **可追溯**：每条覆盖可追溯到具体 run 与 commit。

## 4. 存储设计（混合存储方案）

### 4.1 存储策略概述

为了平衡查询性能、存储成本和实现复杂度，采用**混合存储策略**：

- **OSS 存储**：存储原始数据和行级覆盖快照（低成本、大容量）
- **数据库存储**：存储聚合统计和元数据（快速查询、支持复杂筛选）

**核心设计原则**：

1. **行级明细不入库**：coverage_snapshot 不存数据库，避免数据量爆炸
2. **快照文件作为唯一真实数据源**：存储在 OSS，JSON.gz 格式
3. **数据库只存聚合统计**：Package/Class/Method 级别，支撑列表展示
4. **按需加载源码详情**：前端查看源码时，从 OSS 下载快照文件解析

**存储规模对比**（以 10 万可执行行的中型项目为例）：

| 存储方案 | 数据库存储 | OSS 存储 | 总成本 |
|---------|-----------|---------|--------|
| 方案 A：数据库存行级明细 | 2 GB (100 个 set) | 500 MB (exec/xml) | 高 |
| 方案 B：混合存储（推荐） | 160 MB (100 个 set) | 1 GB (exec/xml/snapshot) | 低（节省 80%+） |

### 4.2 OSS 存储设计

#### 4.2.1 目录结构

```
{bucket}/reports/
  {coverage_set_id}/
    runs/
      {run_id}/
        jacoco.exec.gz          # 原始采集数据（可选，用于重算）
        jacoco.xml.gz           # 单次覆盖报告（必须）
        meta.json               # 元数据（可选）

    snapshots/
      {commit_id}/
        coverage_snapshot.json.gz   # 累计覆盖快照（核心）
        coverage_stats.json         # 聚合统计（冗余备份，可选）
```

**说明**：

- `runs/` 目录：存储每次 run 的原始数据，用于追溯和重算
- `snapshots/` 目录：存储每个 commit 的累计快照，只保留最新 commit
- `coverage_set_id`：全局唯一标识，建议使用 UUID 或 `{git_url_hash}_{branch}_{scope}` 格式

#### 4.2.2 快照文件格式（coverage_snapshot.json.gz）

**文件格式**：JSON + GZIP 压缩

**为什么选择 JSON.gz**：
- 文本格式，压缩比高（通常压缩到原始大小的 10-20%）
- Java 解析简单，生态成熟（Jackson、Gson）
- 可读性好，便于调试和排查问题
- 支持流式解析，大文件也能处理（使用 JsonParser）

**文件结构**：

```json
{
  "metadata": {
    "coverage_set_id": "set_abc123",
    "commit_id": "abc123def456",
    "git_url": "https://github.com/example/repo.git",
    "branch": "master",
    "generated_at": "2026-02-25T09:00:00Z",
    "generator_version": "super-jacoco-2.0.0"
  },

  "summary": {
    "total_files": 150,
    "total_lines": 100000,
    "covered_lines": 80000,
    "missed_lines": 20000,
    "line_coverage_rate": 0.80,
    "total_branches": 5000,
    "covered_branches": 4000,
    "branch_coverage_rate": 0.80
  },

  "files": {
    "src/main/java/com/example/Foo.java": {
      "lines": {
        "1": {"status": "COVERED", "hits": 5, "hash": "a1b2c3d4"},
        "4": {"status": "MISSED", "hits": 0, "hash": "e5f6g7h8"},
        "6": {"status": "COVERED", "hits": 3, "hash": "i9j0k1l2"},
        "10": {"status": "PARTIAL", "hits": 2, "hash": "m3n4o5p6"}
      },
      "summary": {
        "total_lines": 50,
        "covered_lines": 40,
        "line_coverage_rate": 0.80
      }
    },
    "src/main/java/com/example/Bar.java": {
      "lines": {...},
      "summary": {...}
    }
  },

  "packages": {
    "com.example": {
      "line_missed": 200,
      "line_covered": 800,
      "branch_missed": 50,
      "branch_covered": 200
    }
  },

  "classes": {
    "com.example.Foo": {
      "line_missed": 10,
      "line_covered": 40,
      "branch_missed": 5,
      "branch_covered": 20
    }
  }
}
```

**字段说明**：

- `metadata`：快照元数据，用于标识和追溯
- `summary`：全局统计摘要，便于快速获取覆盖率
- `files`：文件级覆盖明细（核心数据）
  - `lines`：行级覆盖信息，key 为行号（字符串格式，便于 JSON 解析）
    - `status`：覆盖状态（COVERED/MISSED/PARTIAL/NO_CODE）
    - `hits`：命中次数（用于合并时取最大值）
    - `hash`：行内容 hash（SHA-256 前 8 字节，用于 diff 继承时精确匹配）
  - `summary`：文件级统计摘要
- `packages`：Package 级聚合统计（冗余存储，便于快速查询）
- `classes`：Class 级聚合统计（冗余存储，便于快速查询）

**行内容 hash 计算方式**：

```java
// 去除行首尾空白后计算 SHA-256，取前 8 字节转 hex
String lineContent = line.trim();
byte[] hash = MessageDigest.getInstance("SHA-256").digest(lineContent.getBytes(UTF_8));
String lineHash = Hex.encodeHexString(Arrays.copyOf(hash, 8));
```

#### 4.2.3 文件大小估算

以 10 万可执行行的中型项目为例：

```
未压缩 JSON 大小：
- 每行约 80 字节（行号 + status + hits + hash + JSON 格式开销）
- 总大小：100,000 * 80 = 8 MB

GZIP 压缩后：
- 压缩比约 10-20%
- 压缩后大小：0.8 - 1.6 MB

结论：单个快照文件约 1-2 MB，存储成本极低
```

#### 4.2.4 生命周期管理

**保留策略**：

- `runs/` 目录：保留最近 30 天的 run 数据，超过 30 天自动删除或归档
- `snapshots/` 目录：只保留最新 commit 的快照，commit 变化时覆盖旧快照

**归档策略**（可选）：

- 对于已归档的 coverage_set（status=ARCHIVED），可将快照文件转移到归档存储（如 S3 Glacier）
- 归档后的快照文件仍可按需恢复查看

### 4.3 数据库存储设计

数据库只存储**元数据**和**聚合统计**，不存储行级明细。

#### 4.3.1 存储内容

1. **元数据表**：coverage_set、coverage_run、report_artifact
2. **聚合统计表**：coverage_node_stats（Package/Class/Method 级）
3. **不存储**：~~coverage_snapshot~~（行级明细存 OSS）

#### 4.3.2 数据量估算

以 100 个 coverage_set 为例：

```
coverage_set 表：100 行
coverage_run 表：100 * 30（每个 set 平均 30 次 run）= 3,000 行
report_artifact 表：3,000 * 2（每次 run 2 个文件）= 6,000 行
coverage_node_stats 表：100 * 5,500（每个 set 平均 5,500 个节点）= 550,000 行

总行数：约 56 万行
总存储：约 160 MB（远小于行级明细的 2 GB）
```

### 4.4 缓存策略

为了提升查询性能，采用多级缓存：

#### 4.4.1 Redis 缓存（可选）

**缓存内容**：
- 热点 coverage_set 的快照文件（完整 JSON）
- 热点文件的行级覆盖数据

**缓存策略**：
- Key 格式：`cov:snapshot:{coverage_set_id}:{commit_id}`
- TTL：1 小时
- 淘汰策略：LRU

**缓存命中率预期**：
- 列表页查询：100%（查数据库，不走缓存）
- 源码详情页：首次访问 0%，后续访问 80%+

#### 4.4.2 本地内存缓存

**缓存内容**：
- 最近访问的文件级覆盖数据（Map<String, FileLineCoverage>）

**缓存策略**：
- 使用 Caffeine 或 Guava Cache
- 最大容量：100 个文件
- TTL：10 分钟
- 淘汰策略：LRU

#### 4.4.3 HTTP 缓存

**前端缓存**：
- 列表页 API：Cache-Control: max-age=300（5 分钟）
- 源码详情 API：Cache-Control: max-age=3600（1 小时）
- 快照文件不变时，使用 ETag 实现 304 Not Modified

## 5. 数据模型

### 5.1 coverage_campaign（可选）

用于组织多个 coverage_set，支持"覆盖活动"概念。

```sql
CREATE TABLE coverage_campaign (
  campaign_id VARCHAR(80) PRIMARY KEY,
  campaign_name VARCHAR(256) NOT NULL,
  git_url VARCHAR(512) NOT NULL,
  branch VARCHAR(256) NOT NULL,
  metric_scope VARCHAR(32) NOT NULL COMMENT 'FULL/DIFF',
  from_type VARCHAR(32) NOT NULL COMMENT 'UNIT/ENV/LOCAL',
  description TEXT,
  status VARCHAR(32) NOT NULL DEFAULT 'ACTIVE' COMMENT 'ACTIVE/ARCHIVED',
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX idx_git_branch (git_url(255), branch(100))
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 5.2 coverage_set（必须）

覆盖集合，绑定到"服务 + 分支 + 口径"，允许跨 commit 累计。

```sql
CREATE TABLE coverage_set (
  coverage_set_id VARCHAR(80) PRIMARY KEY,
  campaign_id VARCHAR(80) COMMENT '所属活动，可空',
  git_url VARCHAR(512) NOT NULL,
  branch VARCHAR(256) NOT NULL,
  type VARCHAR(32) NOT NULL COMMENT 'FULL/DIFF',
  from_type VARCHAR(32) NOT NULL COMMENT 'UNIT/ENV/LOCAL',
  scope_key VARCHAR(512) NOT NULL COMMENT '唯一标识：git_url+branch+type+from',
  current_commit_id VARCHAR(80) COMMENT '当前最新 commit',
  current_snapshot_key VARCHAR(1024) COMMENT '当前快照文件 OSS key',
  line_coverage_rate DECIMAL(5,4) COMMENT '当前行覆盖率',
  branch_coverage_rate DECIMAL(5,4) COMMENT '当前分支覆盖率',
  status VARCHAR(32) NOT NULL DEFAULT 'ACTIVE' COMMENT 'ACTIVE/ARCHIVED',
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_scope (scope_key(255)),
  INDEX idx_git_branch (git_url(255), branch(100)),
  INDEX idx_status (status),
  FOREIGN KEY (campaign_id) REFERENCES coverage_campaign(campaign_id) ON DELETE SET NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 5.3 coverage_run（必须）

每次覆盖采集记录。

```sql
CREATE TABLE coverage_run (
  run_id VARCHAR(80) PRIMARY KEY,
  coverage_set_id VARCHAR(80) NOT NULL,
  commit_id VARCHAR(80) NOT NULL,
  commit_message TEXT COMMENT 'commit 信息，便于追溯',
  exec_object_key VARCHAR(1024) COMMENT 'jacoco.exec.gz 的 OSS key',
  xml_object_key VARCHAR(1024) NOT NULL COMMENT 'jacoco.xml.gz 的 OSS key',
  line_coverage_rate DECIMAL(5,4) COMMENT '本次覆盖率',
  branch_coverage_rate DECIMAL(5,4),
  status VARCHAR(32) NOT NULL DEFAULT 'PENDING' COMMENT 'PENDING/PROCESSING/COMPLETED/FAILED',
  error_message TEXT COMMENT '失败原因',
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX idx_set_commit (coverage_set_id, commit_id),
  INDEX idx_created (created_at),
  FOREIGN KEY (coverage_set_id) REFERENCES coverage_set(coverage_set_id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 5.4 coverage_node_stats（新增，核心）

Package/Class/Method 级聚合统计，支撑前端列表展示。

```sql
CREATE TABLE coverage_node_stats (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,

  display_name VARCHAR(1024) NOT NULL COMMENT '显示名称',
  source_file VARCHAR(1024) COMMENT '源文件路径（CLASS/METHOD 有值）',
  line_missed INT NOT NULL DEFAULT 0,
  line_covered INT NOT NULL DEFAULT 0,
  branch_missed INT NOT NULL DEFAULT 0,
  branch_covered INT NOT NULL DEFAULT 0,
  instruction_missed INT NOT NULL DEFAULT 0,
  instruction_covered INT NOT NULL DEFAULT 0,
  complexity_missed INT NOT NULL DEFAULT 0,
  complexity_covered INT NOT NULL DEFAULT 0,
  method_missed INT NOT NULL DEFAULT 0,
  method_covered INT NOT NULL DEFAULT 0,
  class_missed INT NOT NULL DEFAULT 0,
  class_covered INT NOT NULL DEFAULT 0,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
```

## 7. 工作流程详细说明

### 7.1 端到端流程概览

```
1. 采集阶段
   ├─ 测试执行，生成 jacoco.exec
   ├─ 生成 jacoco.xml.gz
   └─ 上传 OSS + 记录 coverage_run

2. 解析阶段（异步）
   ├─ 从 OSS 下载 xml.gz
   ├─ 解析成内存中的行级快照
   └─ 判断 commit 是否变化

3. 继承阶段（commit 变化时）
   ├─ 从 OSS 下载上一个 commit 的累计快照
   ├─ 执行 Git diff
   ├─ 根据 diff 结果做继承映射
   └─ 生成继承快照

4. 合并阶段
   ├─ 合并新覆盖到继承快照（OR 操作）
   ├─ 生成新的累计快照
   └─ 上传 OSS

5. 聚合阶段
   ├─ 聚合生成 Package/Class/Method 统计
   ├─ 写入/更新 coverage_node_stats 表
   └─ 更新 coverage_set 表
```

### 7.2 Git diff 命令与解析字段说明

#### 推荐命令

```bash
git diff --find-renames --unified=0 <old_commit> <new_commit> -- <path>
```

**参数说明**：
- `--find-renames`：识别文件重命名，避免误判为"删除+新增"
- `--unified=0`：只输出变更行，不输出上下文，便于行级映射

#### 解析字段（核心）

- `diff --git a/... b/...`：文件对比开始，包含旧新路径
- `rename from ... / rename to ...`：文件重命名映射
- `@@ -old_start,old_len +new_start,new_len @@`：行号映射区间
  - `old_start/old_len`：旧文件变更段起始行与长度
  - `new_start/new_len`：新文件变更段起始行与长度
- 变更行前缀：
  - `-`：旧文件删除行（新文件不存在）
  - `+`：新文件新增行（旧文件不存在）

#### 行级映射处理要点

- 对未出现在 diff 中的行，默认视为 unchanged，可继承覆盖
- 对 `-`/`+` 标记的行，按"删除/新增"处理
- 对重命名文件，先建立路径映射，再进行行级继承

```java
    // 1. 查询 run 信息
    CoverageRun run = coverageRunMapper.selectById(runId);

    // 2. 从 OSS 下载 xml.gz
    InputStream xmlStream = ossClient.getObject(run.getXmlObjectKey());

    // 3. 解析成内存中的行级快照
    CoverageSnapshot newSnapshot = snapshotGenerator.generateFromXml(
        xmlStream, run.getCommitId()
    );

    // 4. 判断 commit 是否变化
    CoverageSet set = coverageSetMapper.selectById(run.getCoverageSetId());
    boolean commitChanged = !run.getCommitId().equals(set.getCurrentCommitId());

    if (commitChanged) {
        // commit 变化，需要 diff 继承
        processCommitChange(set, run, newSnapshot);
    } else {
        // commit 未变化，直接合并
        mergeSnapshot(set, newSnapshot);
    }
}
```

```java
private void processCommitChange(CoverageSet set, CoverageRun run,
                                  CoverageSnapshot newSnapshot) {
    // 1. 从 OSS 下载上一个 commit 的累计快照
    CoverageSnapshot oldSnapshot = downloadSnapshot(
        set.getCurrentSnapshotKey()
    );

    // 2. 执行 Git diff
    GitDiffResult diff = gitService.diff(
        set.getCurrentCommitId(),  // old commit
        run.getCommitId(),          // new commit
        set.getGitUrl()
    );

    // 3. 根据 diff 结果做继承映射
    CoverageSnapshot inheritedSnapshot = snapshotGenerator.inherit(
        oldSnapshot,
        set.getCurrentCommitId(),
        run.getCommitId(),
        diff
    );

    // 4. 合并新覆盖到继承快照
    CoverageSnapshot mergedSnapshot = snapshotGenerator.merge(
        inheritedSnapshot,
        newSnapshot
    );

    // 5. 上传新快照到 OSS
    String newSnapshotKey = uploadSnapshot(mergedSnapshot, set.getCoverageSetId());

    // 6. 更新 coverage_set
    set.setCurrentCommitId(run.getCommitId());
    set.setCurrentSnapshotKey(newSnapshotKey);
    coverageSetMapper.updateById(set);

    // 7. 聚合统计并写入数据库
    aggregateAndSave(set, mergedSnapshot);
}
```

### 7.4 diff 继承算法详解

#### 算法伪代码

```java
public CoverageSnapshot inherit(CoverageSnapshot oldSnapshot,
                                 String oldCommit,
                                 String newCommit,
                                 GitDiffResult diff) {
    CoverageSnapshot inherited = new CoverageSnapshot();
    inherited.setCommitId(newCommit);

    // 遍历旧快照的每个文件
    for (String filePath : oldSnapshot.getFiles().keySet()) {
        FileCoverage oldFile = oldSnapshot.getFile(filePath);

        // 检查文件是否被删除
        if (diff.isFileDeleted(filePath)) {
            continue; // 文件删除，不继承
        }

        // 检查文件是否被重命名
        String newFilePath = diff.getNewPath(filePath);
        if (newFilePath == null) {
            newFilePath = filePath; // 未重命名
        }

        // 获取该文件的行级 diff
        FileDiff fileDiff = diff.getFileDiff(filePath);

        FileCoverage newFile = new FileCoverage();

        // 遍历旧文件的每一行
        for (Map.Entry<Integer, LineCoverage> entry : oldFile.getLines().entrySet()) {
            int oldLineNo = entry.getKey();
            LineCoverage oldLine = entry.getValue();

            // 检查该行是否变更
            LineMapping mapping = fileDiff.getLineMapping(oldLineNo);

            if (mapping.isUnchanged()) {
                // 未变更，继承覆盖状态
                int newLineNo = mapping.getNewLineNo();
                newFile.addLine(newLineNo, oldLine);

            } else if (mapping.isMoved()) {
                // 行号移动，通过 line_hash 二次校验
                if (oldLine.getHash().equals(mapping.getNewLineHash())) {
                    // hash 相同，确认是移动而非修改，继承覆盖
                    int newLineNo = mapping.getNewLineNo();
                    newFile.addLine(newLineNo, oldLine);
                } else {
                    // hash 不同，说明内容变更，不继承
                }

            } else {
                // 行被删除或修改，不继承
            }
        }

        inherited.addFile(newFilePath, newFile);
    }

    return inherited;
}
```

### 8.1 快照生成器实现

#### CoverageSnapshotGenerator.java

```java
public class CoverageSnapshotGenerator {

    /**
     * 从 jacoco.xml 生成覆盖快照
     */
    public CoverageSnapshot generateFromXml(InputStream xmlStream, String commitId) {
        CoverageSnapshot snapshot = new CoverageSnapshot();
        snapshot.setCommitId(commitId);

        try {
            SAXParserFactory factory = SAXParserFactory.newInstance();
            SAXParser parser = factory.newSAXParser();

            parser.parse(xmlStream, new DefaultHandler() {
                private String currentPackage;
                private String currentClass;
                private String currentSourceFile;

                @Override
                public void startElement(String uri, String localName,
                                        String qName, Attributes attrs) {
                    if ("package".equals(qName)) {
                        currentPackage = attrs.getValue("name");
                    } else if ("class".equals(qName)) {
                        currentClass = attrs.getValue("name");
                        currentSourceFile = attrs.getValue("sourcefilename");
                    } else if ("line".equals(qName)) {
                        int lineNo = Integer.parseInt(attrs.getValue("nr"));
                        int missed = Integer.parseInt(attrs.getValue("mi"));
                        int covered = Integer.parseInt(attrs.getValue("ci"));

                        LineCoverage line = new LineCoverage();
                        line.setLineNo(lineNo);
                        line.setStatus(covered > 0 ? "COVERED" : "MISSED");
                        line.setHits(covered);

                        String filePath = currentPackage.replace('.', '/')
                                        + "/" + currentSourceFile;
                        snapshot.addLine(filePath, lineNo, line);
                    }
                }
            });
        } catch (Exception e) {
            throw new RuntimeException("Failed to parse jacoco.xml", e);
        }

        return snapshot;
    }

    /**
     * 合并两个快照（OR 操作）
     */
    public CoverageSnapshot merge(CoverageSnapshot base, CoverageSnapshot delta) {
        CoverageSnapshot merged = new CoverageSnapshot();
        merged.setCommitId(delta.getCommitId());

        // 先复制 base 的所有覆盖
        for (String filePath : base.getFiles().keySet()) {
            merged.putFile(filePath, base.getFile(filePath).copy());
        }

        // 合并 delta 的覆盖
        for (String filePath : delta.getFiles().keySet()) {
            FileCoverage deltaFile = delta.getFile(filePath);
            FileCoverage mergedFile = merged.getFile(filePath);

            if (mergedFile == null) {
                merged.putFile(filePath, deltaFile.copy());
            } else {
                for (Map.Entry<Integer, LineCoverage> entry :
                     deltaFile.getLines().entrySet()) {
                    int lineNo = entry.getKey();
                    LineCoverage deltaLine = entry.getValue();
                    LineCoverage baseLine = mergedFile.getLine(lineNo);

                    if (baseLine == null) {
                        mergedFile.addLine(lineNo, deltaLine);
                    } else {
                        // 合并：只要有一个覆盖就算覆盖，hits 取最大值
                        if ("COVERED".equals(deltaLine.getStatus()) ||
                            "COVERED".equals(baseLine.getStatus())) {
                            baseLine.setStatus("COVERED");
                            baseLine.setHits(Math.max(
                                baseLine.getHits(), deltaLine.getHits()
                            ));
                        }
                    }
                }
            }
        }

        return merged;
    }
}
```

### 8.2 Git Diff 解析器实现

#### GitDiffParser.java

```java
public class GitDiffParser {

    /**
     * 执行 git diff 并解析结果
     */
    public GitDiffResult diff(String gitUrl, String oldCommit,
                              String newCommit, String localRepoPath) {
        // 1. 确保本地仓库是最新的
        ensureRepoUpdated(gitUrl, localRepoPath);

        // 2. 执行 git diff
        String diffOutput = execGitDiff(localRepoPath, oldCommit, newCommit);

        // 3. 解析 diff 输出
        return parseDiffOutput(diffOutput);
    }

    private String execGitDiff(String repoPath, String oldCommit,
                               String newCommit) {
        ProcessBuilder pb = new ProcessBuilder(
            "git", "diff",
            "--find-renames",
            "--unified=0",
            oldCommit,
            newCommit
        );
        pb.directory(new File(repoPath));

        try {
            Process process = pb.start();
            String output = IOUtils.toString(
                process.getInputStream(), StandardCharsets.UTF_8
            );
            process.waitFor();
            return output;
        } catch (Exception e) {
            throw new RuntimeException("Failed to execute git diff", e);
        }
    }

    private GitDiffResult parseDiffOutput(String diffOutput) {
        GitDiffResult result = new GitDiffResult();
        String[] lines = diffOutput.split("\n");

        String currentOldPath = null;
        String currentNewPath = null;
        FileDiff currentFileDiff = null;

        for (String line : lines) {
            if (line.startsWith("diff --git")) {
                // 新文件开始
                String[] parts = line.split(" ");
                currentOldPath = parts[2].substring(2); // 去掉 a/
                currentNewPath = parts[3].substring(2); // 去掉 b/
                currentFileDiff = new FileDiff(currentOldPath, currentNewPath);
                result.addFileDiff(currentFileDiff);

            } else if (line.startsWith("rename from")) {
                currentFileDiff.setRenamed(true);

            } else if (line.startsWith("@@")) {
                // 行号映射区间
                parseHunkHeader(line, currentFileDiff);
            }
        }

        return result;
    }

    private void parseHunkHeader(String line, FileDiff fileDiff) {
        // @@ -old_start,old_len +new_start,new_len @@
        Pattern pattern = Pattern.compile(
            "@@ -(\\d+)(?:,(\\d+))? \\+(\\d+)(?:,(\\d+))? @@"
        );
        Matcher matcher = pattern.matcher(line);

        if (matcher.find()) {
            int oldStart = Integer.parseInt(matcher.group(1));
            int oldLen = matcher.group(2) != null ?
                        Integer.parseInt(matcher.group(2)) : 1;
            int newStart = Integer.parseInt(matcher.group(3));
            int newLen = matcher.group(4) != null ?
                        Integer.parseInt(matcher.group(4)) : 1;

            // 记录变更区间
            fileDiff.addChangedRange(oldStart, oldLen, newStart, newLen);
        }
    }
}
```

## 9. 风险与应对

### 9.1 技术风险

**风险 1：diff 映射错误导致少量误继承**

- **描述**：Git diff 在复杂重构场景下可能出现误判，导致变更行被误认为未变更
- **影响**：旧覆盖被错误继承到新代码，覆盖率虚高
- **应对**：
  - 行级 diff + line_hash 二次校验
  - 对于大规模重构（如整个类重命名），提供手动触发"重置覆盖"功能
  - 监控覆盖率异常波动，及时发现问题

**风险 2：快照文件体量较大**

- **描述**：大型项目的快照文件可能达到 10+ MB（压缩后）
- **影响**：OSS 下载耗时增加，影响查询性能
- **应对**：
  - 快照文件按文件拆分存储（可选优化）
  - 使用 CDN 加速 OSS 访问
  - 本地缓存热点快照文件

**风险 3：异步处理延迟**

- **描述**：run 提交后，异步处理可能需要 10-30 秒
- **影响**：用户提交后立即查看，看不到最新数据
- **应对**：
  - 前端轮询 run 状态，处理完成后自动刷新
  - 提供"处理中"状态提示
  - 优化处理流程，缩短延迟

### 9.2 数据风险

**风险 4：OSS 文件丢失**

- **描述**：OSS 故障或误删除导致快照文件丢失
- **影响**：无法查看历史覆盖详情
- **应对**：
  - OSS 开启版本控制和回收站
  - 定期备份关键快照文件
  - 数据库保留聚合统计，即使快照丢失也能看到摘要

**风险 5：数据库存储压力**

- **描述**：coverage_node_stats 表随着 set 数量增长可能达到百万级
- **影响**：查询性能下降
- **应对**：
  - 定期归档历史数据（status=ARCHIVED 的 set）
  - 分表策略（按年份或按项目）
  - 索引优化

## 10. 推进方式（分阶段交付）

### 阶段 1：基础能力（2-3 周）

**目标**：支持单 commit 的覆盖累积更新

**交付内容**：
1. 数据模型建表（coverage_set、coverage_run、coverage_node_stats、report_artifact）
2. OSS 存储集成（上传 xml.gz、exec.gz）
3. 快照生成器（从 xml 解析成 JSON 快照）
4. 基础 API（创建 set、追加 run、查询概览）

**验收标准**：
- 同一 commit 下，多次 run 的覆盖能够合并
- 覆盖率随测试补齐逐步提升

### 阶段 2：跨 commit 继承（3-4 周）

**目标**：支持代码更新后的覆盖继承

**交付内容**：
1. Git diff 解析器
2. diff 继承算法实现
3. commit 变化检测与自动触发
4. 前端展示优化（显示 commit 信息）

**验收标准**：
- commit 变化时，未变更代码的覆盖能够继承
- 变更代码的覆盖清零，需要新测试补齐
- 覆盖率变化可解释（继承 + 新增）

### 阶段 3：性能优化与完善（2-3 周）

**目标**：生产环境稳定运行

**交付内容**：
1. 缓存策略（Redis + 本地缓存）
2. 异步处理优化（消息队列）
3. 监控告警（处理延迟、失败率、覆盖率异常波动）
4. 前端体验优化（加载状态、错误提示）

**验收标准**：
- 列表页查询 < 100ms
- 源码详情页首次加载 < 2s，缓存命中 < 100ms
- 异步处理成功率 > 99%
