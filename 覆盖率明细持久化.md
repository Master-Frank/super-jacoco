# 覆盖率明细持久化（纯 XML 制品 + API 驱动）方案

## 1. 目标与边界

### 目标

- 报告文件被清理/丢失后，仍可获取完整覆盖率明细（Project -> Package -> Class -> Method -> Line）。
- **完全解耦前端展示与后端存储**：不依赖原生 HTML 报告，前端 UI 通过 API 获取结构化数据渲染。
- 低改造成本、可渐进式上线；使用对象存储作为唯一数据源。

### 约束与现状

- 现状数据库仅存摘要与任务元数据：`diff_coverage_report`（line/branch 覆盖率、report_url、diffmethod、状态等）。
- 现状报告产物在磁盘：`cov.paths.reportRoot/{uuid}/index.html` 以及同目录 `jacoco.xml`。
- 任务频率：每天约 3 次；项目数量：100+；估算每天 300 份报告。
- **决策变更**：不再保留/上传原生 HTML 报告，统一采用 XML 解析入库 + API 方案。

### 方案核心思路

- **对象存储（OSS/COS/S3）**：只保存 `jacoco.xml.gz`（作为唯一事实数据源）和 `meta.json`。
- **数据库**：
  - `report_artifact`：索引 OSS 上的 XML 文件位置。
  - `coverage_node_stats`：异步解析 XML 后存入的结构化统计数据（Package/Class/Method 级），支撑列表与概览页**秒开**。
- **按需解析**：源码级（Line）明细不全量入库，前端查看详情时后端实时从 XML 提取并缓存。

## 2. 总体架构

### 组件

- 覆盖率服务（super-jacoco）：生成 `jacoco.xml`，计算 line/branch 百分比。
- 对象存储：存放 `jacoco.xml.gz` 制品。
- MySQL：存放任务记录、制品索引、结构化明细聚合（Package/Class/Method）。
- （可选）Redis：缓存热门 uuid 的源码级解析结果（Line 级）。
- （建议）消息队列/异步任务池：用于解耦“报告生成”与“XML 解析入库”，避免长时阻塞。

### 访问路径

- **列表/概览/下钻**：前端 -> 覆盖率服务 API -> MySQL (`coverage_node_stats`)
  - *特点：数据预计算，毫秒级响应，支持排序/筛选。*
- **源码行级详情**：前端 -> 覆盖率服务 API ->（服务端从 OSS 拉取 xml.gz）-> 解析 -> 返回 JSON
  - *特点：按需加载，首次访问约 1s，命中缓存后毫秒级。*

## 3. 对象存储（制品）设计

### 3.1 Bucket 与权限

- 建议使用 **私有桶**。
- CORS 配置：若前端需直接从浏览器 AJAX 拉取 XML（非推荐，建议走后端 API 转发），需配置 CORS。**本方案推荐走后端 API**，无需配置 CORS。

### 3.2 Key 命名规范（强烈建议固定）

以 `projectKey + 任务类型 + 日期 + uuid` 组织：

```
reports/
  {projectKey}/
    {from}/            # UNIT | ENV
      {type}/          # FULL | DIFF
        {yyyy}/{MM}/{dd}/
          {uuid}/
            meta.json
            jacoco.xml.gz
```

说明：

- `projectKey`：建议引入（如 gitUrl 的仓库名或业务自定义 key），用于后续按项目检索与生命周期策略。
- `meta.json`：存最关键的可读信息（摘要、时间、版本、覆盖率、生成器版本等）。

### 3.3 制品内容建议

- **必存**：`jacoco.xml.gz`
  - 核心数据源，体积小（文本压缩比高），包含所有层级覆盖数据。
- **建议存**：`meta.json`
  - 便于排障、快速定位、批量校验。
- **可选存**：`jacoco.exec.gz`
  - 仅当需要未来“更换过滤规则重算”时才存。

### 3.4 生命周期与成本控制

针对你当前量级（300 份/天），推荐：

- XML 数据保留较久（例如 180 天或 1 年），以支持长期趋势分析。
- 180 天后转归档存储。

## 4. 数据库（结构化索引）设计

### 4.1 现有表（无需变更即可用）

`diff_coverage_report` 保持不变，用于存任务状态和最顶层摘要。

### 4.2 新增表：report_artifact（制品索引）

一条 uuid 对应 1-2 条制品记录（XML/META）。

字段建议：

- `id` BIGINT PK
- `job_record_uuid` VARCHAR(80) NOT NULL
- `artifact_type` VARCHAR(32) NOT NULL  # XML_GZ | META_JSON
- `bucket` VARCHAR(128) NOT NULL
- `object_key` VARCHAR(1024) NOT NULL
- `size_bytes` BIGINT NOT NULL DEFAULT 0
- `created_at` TIMESTAMP
- 索引：(`job_record_uuid`, `artifact_type`)

### 4.3 新增表：coverage_node_stats（结构化明细聚合）

用于支撑前端“按包/类/方法”的明细页面与聚合排行。
**仅存 Package / Class / Method 三个层级**。

字段建议（示例）：

- `id` BIGINT PK
- `job_record_uuid` VARCHAR(80) NOT NULL
- `node_type` VARCHAR(16) NOT NULL     # PACKAGE | CLASS | METHOD
- `node_key` VARCHAR(1024) NOT NULL    # 全限定名，如 com.a.b | com.a.C | com.a.C#foo
- `node_level` TINYINT DEFAULT 0       # 0=PACKAGE, 1=CLASS, 2=METHOD
- `display_name` VARCHAR(1024) NOT NULL
- `parent_key` VARCHAR(1024) DEFAULT ''
- `line_missed` INT NOT NULL DEFAULT 0
- `line_covered` INT NOT NULL DEFAULT 0
- `branch_missed` INT NOT NULL DEFAULT 0
- `branch_covered` INT NOT NULL DEFAULT 0
- （...其他 counter 指标...）
- 索引：(`job_record_uuid`, `node_level`), (`job_record_uuid`, `parent_key`)

### 4.4 可选表：coverage_line_cache（逐行明细缓存）

用于缓存“源码详情页”的解析结果。

- `job_record_uuid`
- `class_key` (如 com.example.MyClass)
- `content_json` (行覆盖状态的 JSON，如 `{1: "COVERED", 2: "MISSED"}`)
- `expire_at`

## 5. 数据流（可执行步骤）

### 阶段 1：制品上云与索引

1) 任务结束生成 `jacoco.xml`。
2) 压缩为 `jacoco.xml.gz` 并上传 OSS。
3) 写入 `report_artifact` 表。
4) 清理本地磁盘文件。

### 阶段 2：异步解析与聚合入库

1) 触发异步任务（MQ 或 线程池）。
2) 从 OSS 拉取 `jacoco.xml.gz`。
3) SAX 流式解析 XML：
   - 遍历 Package/Class/Method 节点，累加 Counters。
   - 组装 `coverage_node_stats` 数据对象。
4) 批量插入 DB（先删后插，保证幂等）。

### 阶段 3：API 服务化

1) **列表/下钻 API**：查询 `coverage_node_stats` 表。
2) **详情 API**：
   - 接收 `uuid` + `classKey`。
   - 查缓存 -> 未命中则拉取 OSS XML -> 解析该 Class 的 `<line>` 标签 -> 返回 JSON -> 写入缓存。

## 6. 前端渲染建议

前端完全基于 JSON 数据渲染，不再嵌入 IFrame。

### 页面 1：概览与包列表
- API: `GET /api/cov/{uuid}/nodes?level=PACKAGE`
- UI: 表格展示所有包的覆盖率，支持排序。

### 页面 2：类列表（包详情）
- API: `GET /api/cov/{uuid}/nodes?level=CLASS&parentKey=com.example`
- UI: 展示该包下所有类的覆盖率。

### 页面 3：源码详情（行覆盖）
- API: `GET /api/cov/{uuid}/source?classKey=com.example.Demo`
- Response:
  ```json
  {
    "source": "public class Demo { ... }",  // 可选：如果 XML 不含源码，需从 git 拉取或前端不展示源码仅展示行号状态
    "lines": {
      "12": { "status": "COVERED", "hits": 5 },
      "13": { "status": "MISSED", "hits": 0 }
    }
  }
  ```
- UI: 使用 Monaco Editor 或 highlight.js 渲染代码，根据 `lines` 数据在行号旁标记红/绿颜色。

## 7. 运维与监控

- **解析延迟**：监控“任务结束”到“DB 聚合数据可见”的时间差（通常应 < 10s）。
- **缓存命中率**：监控源码详情页的缓存命中情况。

## 8. 存量迁移

- 遍历磁盘旧报告目录 -> 找到 jacoco.xml -> 压缩上传 OSS -> 触发解析入库 -> 删除磁盘文件。
- 这是一个一次性脚本任务，完成后即可彻底废弃本地报告存储。
