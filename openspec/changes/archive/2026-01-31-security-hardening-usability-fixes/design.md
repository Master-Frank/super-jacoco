## Context

该仓库是一个基于 Spring Boot 的覆盖率收集服务，能够通过接口触发拉取代码、编译、从目标服务 dump jacoco.exec 并生成 HTML 报告。当前实现存在以下系统性问题：

- 接口缺少鉴权与访问控制，但可触发高权限/高成本的外部动作。
- 外部命令执行依赖字符串拼接并通过 `bash -c` 运行，且参数来源包含请求/DB 字段，存在命令注入与跨平台不可用风险。
- 若干可用性缺陷导致“跑不通/结果不准”，例如 Maven profile 参数拼接、单/多模块判断分支不生效。
- 异步执行模型不统一：既有 `new Thread()`，也有自建线程池；任务领取依赖 DB CAS，但缺少一致的并发上限与失败恢复策略。

本变更目标是在不改变核心产品能力（全量/增量覆盖率收集、报告生成与返回）前提下，完成安全收敛与可用性修复。

## Goals / Non-Goals

**Goals:**

- 覆盖率触发接口具备可配置的访问控制（鉴权、并发保护、可选 IP allowlist），默认安全。
- 外部命令执行改造为参数化执行并对输入做白名单校验，降低注入风险并提升可移植性。
- 修复已识别的关键可用性问题（profile 参数、模块判断、日志与异常堆栈）。
- 统一任务异步执行与状态更新策略，确保并发可控、失败可观测、结果可追踪。

**Non-Goals:**

- 不在本变更中升级 Spring Boot 或大规模升级依赖版本。
- 不引入新的分布式调度系统或重写任务模型为完整的队列系统。
- 不改变覆盖率计算口径（全量/增量方法）与现有 API 路径。

## Decisions

1. 访问控制实现方式

- 选择：在 Spring MVC 层引入统一的请求拦截（例如 Filter/Interceptor），对 `/cov/**` 执行 Token 校验，并提供可配置开关。
- 原因：改动集中、侵入小；可在不影响业务逻辑的情况下统一处理鉴权、IP allowlist 与基础限流/并发保护。
- 备选：每个 Controller 方法内显式校验（被拒绝：易遗漏、重复代码多、难统一策略）。

2. 命令执行从 bash 字符串拼接迁移

- 选择：封装一个“受控命令执行器”，使用 `ProcessBuilder(List<String>)` 传参，明确工作目录、超时与输出采集；禁止把整段命令交给 shell 解析。
- 原因：消除 shell 解析带来的注入面与平台差异；更容易对参数做结构化校验与审计。
- 备选：继续使用 `bash -c` 但做转义（被拒绝：仍依赖 shell，转义规则复杂且容易出错）。

3. 输入校验与白名单策略

- 选择：对可能进入命令行/文件系统的输入（uuid、gitUrl、commitId/branch、subModule、路径）定义严格校验规则；对路径采用“只允许在工作根目录下”的约束；对 gitUrl 支持明确的协议/域名白名单。
- 原因：该系统本质上是“把输入转换为外部动作”，输入校验是核心安全边界。

4. 异步与并发控制

- 选择：统一使用一个可配置的线程池执行后台任务，并在任务领取/执行阶段加入并发上限控制；避免 `new Thread()`。
- 原因：可控并发、防止线程泄漏、便于监控与调优。

5. 可用性修复的边界

- 选择：在不重构业务流程的前提下，修复确定会导致失败/错误结果的问题（profile 参数拼接、模块列表判定、日志堆栈输出、异常消息一致性）。
- 原因：该类问题收益高且风险可控。

## Risks / Trade-offs

- [访问控制引入后导致现有调用方失败] → 提供兼容模式（可配置关闭鉴权/仅内网 allowlist），并在发布前与调用方对齐 Token 获取方式。
- [命令执行迁移导致行为差异] → 对关键命令（mvn、java -jar jacoco、文件拷贝/清理）逐一对齐参数与工作目录，保留详细执行日志与退出码。
- [严格输入校验误伤合法参数] → 以白名单为主并逐步收紧；对被拒绝输入返回明确错误码与可读提示。
- [并发限制影响吞吐] → 通过配置化线程池与队列长度调优，在安全默认与可用性之间平衡。
